먼저 이것만 이해하면 된다

Linear의 역할

Linear는 “무슨 일을 할지”를 적고, 작업을 시작/중지/재시도하는 명령판이다. 쉽게 말해 작업 티켓과 대화가 모이는 곳이다.

GitHub의 역할

GitHub는 실제 코드가 바뀌고 PR을 리뷰하는 곳이다. 자동화가 성공하려면 PR 본문이 좋아야 한다.

Notion의 역할

Notion은 결과와 배운 점을 남기는 위키/기억 저장소다. Linear처럼 두 번째 이슈 트래커로 쓰지 않는 것이 좋다.

ReplaceMe의 역할

ReplaceMe는 세 도구 사이를 오케스트레이션한다. 이슈를 읽고, 에이전트를 실행하고, PR을 만들고, 기록을 남긴다.

읽는 법: 아래 7개 아이디어는 구현 계획이 아니라 “어떤 방향으로 제품을 키울지” 고르기 위한 후보들이다. 다음 단계는 마음에 드는 아이디어 하나를 골라 ce-brainstorm으로 요구사항을 구체화하는 것이다.

현재 코드에서 확인한 사실

이미 있는 흐름

현재 ReplaceMe는 API로 티켓을 만들고, Kafka 큐에 넣고, Docker 컨테이너에서 코딩 에이전트를 실행한다. 작업이 끝나면 PR URL, 로그, 상태를 저장한다.

이미 붙어 있는 외부 도구

GitHub/GitLab, Linear/Jira, Notion/Confluence 같은 provider 구조가 있다. 다만 지금은 “깊게 연결된 제품 경험”보다는 “연동 가능한 기본 구조”에 가깝다.

큰 빈틈

Linear는 아직 명령판이라기보다 이슈 생성/댓글 수준이다. GitHub PR 본문은 고정 문장이다. Notion 문서는 수동 endpoint로 만드는 정적 페이지에 가깝다.

외부 흐름

Linear Coding Sessions, GitHub Copilot in Linear, OpenAI Codex in Linear도 모두 “이슈 → 에이전트 → PR” 흐름으로 가고 있다. 이 방향은 시장적으로도 자연스럽다.

생각을 나눈 5가지 축

1. Linear 명령판

이슈를 어떻게 실행 가능한 작업 지시서로 만들까?

2. GitHub PR 리뷰 흐름

에이전트가 만든 PR을 어떻게 믿고 리뷰하게 할까?

3. Notion 기억 저장소

작업 결과와 배운 점을 어떻게 위키에 남길까?

4. 실행 기록과 안전성

무슨 일이 일어났는지 어떻게 추적하고 재시도할까?

5. 개인 환경 설정/건강검사

토큰, 권한, Docker, CLI가 준비됐는지 어떻게 미리 확인할까?

개인 자동화 프로필: personal-github-linear-notion

축 · 개인 설정/건강검사추천도 · 88%복잡도 · 중간

한 줄 설명: “내 개인 개발 자동화 모드”를 하나의 이름 있는 프로필로 만든다.

지금은 GitHub 토큰, Linear API Key, Notion Parent Page, Docker 설정 같은 것들이 흩어져 있다. 이 아이디어는 그것들을 personal-github-linear-notion이라는 하나의 운영 모드로 묶자는 것이다. 이 프로필은 어떤 도구를 쓰는지, 어떤 권한이 필요한지, PR 본문은 어떤 형식인지, Notion에는 어떤 기록을 남길지까지 설명한다.

방향을 보여주는 그림이다. 구현 설계도는 아니다.

근거

현재 코드에는 이미 GitHub/Linear/Notion provider 설정이 있다. 하지만 사용자 경험은 아직 “하나의 개인 자동화 모드”로 묶여 있지 않다.

초보자용 비유

게임에서 캐릭터 프리셋을 고르면 장비와 스킬이 한 번에 맞춰지는 것처럼, 개발 자동화도 “내 개인 GitHub+Linear+Notion 세팅”을 한 번에 고르고 검사할 수 있게 하자는 뜻이다.

단점

처음에는 GitHub+Linear+Notion 조합을 강하게 밀게 된다. 다른 provider도 계속 지원하려면 추상화와 golden path 사이의 균형이 필요하다.

Linear 이슈를 실행 지시서로 만들기

축 · Linear 명령판추천도 · 86%복잡도 · 중간

한 줄 설명: Linear 이슈 하나만 잘 쓰면 ReplaceMe가 바로 작업할 수 있게 만든다.

지금은 API에 repoUrl, baseBranch, 설명 등을 넣어 티켓을 만든다. 앞으로는 Linear 이슈 자체가 작업 지시서가 된다. ReplaceMe는 이슈를 읽고 “이대로 실행해도 안전한가?”를 먼저 확인한다. 정보가 부족하면 바로 실행하지 않고 Linear에 질문 댓글을 남긴다.

방향을 보여주는 그림이다. 실제 UI/명령어 설계는 다음 단계에서 정한다.

근거

현재 코드는 API 티켓을 만들면 바로 큐에 넣는 구조다. Linear 연동은 이슈 생성과 댓글은 가능하지만, 아직 “이슈 자체가 실행 명령”인 구조는 아니다.

왜 중요한가

애매한 이슈를 바로 실행하면 에이전트가 잘못된 코드를 만들 가능성이 높다. 실행 전 질문을 통해 실패 비용을 줄일 수 있다.

단점

어떤 이슈가 “충분히 실행 가능하다”고 볼지 기준을 정해야 한다. 기준이 너무 빡빡하면 사용하기 귀찮아질 수 있다.

3

Run Passport: 실행 기록의 신분증 만들기

축 · 실행 기록과 안전성추천도 · 84%복잡도 · 중간

한 줄 설명: 자동화 한 번이 실행될 때마다 “무슨 일이 있었는지”를 증명하는 기록 카드를 만든다.

Run Passport에는 Linear 이슈, GitHub 브랜치/PR, Notion 문서, 실행한 컨테이너, 사용한 설정, 승인 기록, 로그 요약, 테스트 결과, 실패 이유가 들어간다. Linear 댓글, GitHub PR 본문, Notion 문서는 모두 이 Passport를 가리킨다.

근거

현재 코드에는 Ticket, ExecutionLog, ApprovalRequest가 있어 실행 기록의 재료는 이미 있다. 하지만 이것을 하나의 신뢰 가능한 “실행 영수증”으로 묶는 개념은 아직 없다.

초보자용 비유

택배 송장 번호처럼 생각하면 된다. 송장 번호 하나로 어디서 출발했고, 어디까지 왔고, 문제가 생겼는지 확인할 수 있다. Run Passport는 자동화 작업의 송장 번호다.

단점

로그와 설정 정보를 너무 많이 저장하면 보안 문제가 생길 수 있다. 비밀값과 개인 경로는 반드시 가려야 한다.

4

PR 리뷰 패킷 만들기

축 · GitHub PR 리뷰 흐름추천도 · 82%복잡도 · 중간

한 줄 설명: 에이전트가 PR을 만들 때, 리뷰어가 바로 이해할 수 있는 설명 패키지를 자동으로 작성한다.

지금 GitHub PR 본문은 거의 고정 문장에 가깝다. 앞으로는 PR 본문에 문제, 변경 사항, 테스트, 데모, Linear 이슈 링크, Notion 문서 링크, Run Passport, 남은 위험을 자동으로 넣는다. 이 워크스페이스의 PR 작성 규칙처럼 한국어 4섹션 구조를 기본으로 사용한다.

근거

현재 GitHub PR 생성 코드는 Automated implementation for ticket... 같은 고정 본문을 사용한다. 반면 이 워크스페이스 규칙은 PR/MR 본문을 문제·변경·테스트·데모 구조로 쓰라고 한다.

왜 중요한가

자동화가 만든 PR은 사람이 리뷰해야 한다. PR 본문이 부실하면 자동화가 시간을 아끼기보다 리뷰 시간을 늘린다.

단점

테스트를 실제로 돌리지 않았는데 돌린 것처럼 쓰면 안 된다. 증거가 없으면 “미실행”이라고 솔직히 써야 한다.

5

살아있는 Notion 작업 문서

축 · Notion 기억 저장소추천도 · 80%복잡도 · 중간

한 줄 설명: Notion에 “작업 시작부터 끝까지” 업데이트되는 문서를 만든다.

현재 Notion 문서는 별도 endpoint를 호출해야 만들 수 있고, 내용도 상태/저장소/브랜치/PR 정도의 정적 정보에 가깝다. 앞으로는 작업이 시작되면 Notion 페이지가 생기고, 승인 대기, PR 생성, 실패, 완료, 배운 점까지 계속 업데이트된다. 중요한 실패나 반복되는 패턴은 나중에 다시 참고할 수 있는 위키 조각으로 남긴다.

근거

현재 Notion 연동은 간단한 페이지 생성만 지원하고, Ticket 자체에는 Notion 문서 URL을 계속 추적하거나 업데이트하는 구조가 없다.

왜 중요한가

Notion을 쓰는 이유는 기억을 남기기 위해서다. 단순히 티켓 내용을 복사하는 페이지보다, “왜 이 작업을 했고 무엇을 배웠는지”가 남는 문서가 더 가치 있다.

단점

모든 실행을 전부 Notion에 남기면 위키가 지저분해질 수 있다. 어떤 작업을 위키로 승격할지 기준이 필요하다.

6

Provider Doctor + 로컬 안전문

축 · 개인 설정/건강검사추천도 · 78%복잡도 · 중간

한 줄 설명: 작업 시작 전에 GitHub/Linear/Notion/Docker 준비 상태를 먼저 검사한다.

Provider Doctor는 GitHub 토큰 권한, gh CLI 사용 가능 여부, Linear 팀 접근 권한, Notion 페이지 쓰기 권한, Docker 연결, DB/Kafka 상태, 비밀값 redaction을 확인한다. 로컬 안전문은 “이 API가 외부에 열려도 안전한가?”를 검사한다. 기본은 로컬 개발용으로 안전하게 닫아두는 것이다.

근거

현재 /health는 DB, Kafka, Docker 정도를 확인한다. GitHub/Linear/Notion 권한은 실제 사용 중에야 실패할 가능성이 크다. 문서에도 API 인증/인가와 Docker socket 운영 hardening이 아직 한계로 적혀 있다.

초보자용 비유

배포 전에 “환경변수 다 넣었나? DB 연결되나?” 확인하는 것처럼, 에이전트가 코드를 만지기 전에 외부 도구 권한을 미리 점검하는 안전장치다.

단점

외부 서비스 API와 권한 모델이 바뀌면 Doctor도 업데이트해야 한다. 검사 실패 메시지를 친절하게 만드는 것이 중요하다.

7

실패를 학습으로 바꾸는 루프

축 · 실행 기록과 안전성추천도 · 75%복잡도 · 중간

한 줄 설명: 실패한 자동화 실행을 그냥 버리지 않고, 재시도와 학습 자료로 만든다.

작업이 실패하면 ReplaceMe는 “어디서 실패했는지, 다시 실행해도 되는지, 어떤 정보를 추가해야 하는지”를 보여준다. 필요하면 Linear에 후속 이슈를 만들고, Notion에는 실패 요약을 남긴다. 다음에 비슷한 이슈가 들어오면 예전 실패 기록을 참고해 경고할 수 있다.

근거

현재 실행 로그와 실패 상태는 저장되지만, 재실행 API, Kafka DLQ, 체계적인 retry 정책은 아직 없다. 실패 기록을 Notion 기억과 Run Passport로 연결하면 학습 자산이 된다.

왜 중요한가

개인 자동화는 처음부터 완벽하지 않다. 실패를 잘 남기고 안전하게 다시 시도할 수 있어야 점점 믿을 수 있는 도구가 된다.

단점

자동 재시도는 위험할 수 있다. 처음에는 사용자가 명시적으로 선택하는 “보이는 재시도”가 더 안전하다.

전체 후보를 검토한 결과, 근거가 약하거나 더 큰 아이디어 안에 들어가는 것들은 따로 남기지 않았다.
